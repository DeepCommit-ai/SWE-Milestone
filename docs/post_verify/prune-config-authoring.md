# Post-Verify — Prune Config Authoring & Audit

Residue prune ([spec](../superpowers/specs/residue-prune-spec.md)) is
**default-off**. A dataset must
set `residue_prune: true` explicitly after its src/test partition and prune
facts pass the acceptance gate below. An absent flag preserves the historical
additive overlay, including for datasets that already define a src/test
partition; an explicit `false` also keeps pruning disabled. The mechanism is
universal code, but its correctness is delegated to per-range metadata facts
that **cannot be derived mechanically** — deciding what is source, what is
exam paper, what is environment supply, and what is a fixture-in-disguise is
semantic judgment. Two layers:

| Layer | Where | Decides |
|---|---|---|
| deterministic | `harness/e2e/residue_prune.py` (v2 predicate, START guard, V3b never-delete assertion, fail-closed `scoring_untrusted`) | everything expressible as code over the metadata facts |
| this skill | agent authoring/auditing the metadata facts for each new range | the facts themselves, across any language / repo layout |

If a rule can be written as code over existing fields, put it in the code
layer. This skill exists to author the fields.

## What each field controls (and how it fails)

| Field | Feeds | If wrong |
|---|---|---|
| `repo_src_dirs` | agent-authoritative zone (prune + snapshot capture scope) | too narrow → residue survives + agent work outside scope is silently lost at capture (sidecar `committed_outside` warns); too wide → environment dirs enter the diff surface |
| `test_dirs` | exam-paper protection (`is_test_file`) | a missed pattern lets a GT test be treated as source — V3b aborts the eval rather than delete it, so the failure is loud, but the range is unusable until fixed |
| `exclude_patterns` | carve-outs inside src dirs | missed → vendored/demo trees get pruned as if agent-owned |
| `generated_patterns` | environment supply (`.pb.go`, `wire_gen.go`, bindgen output) | missed → generated file absent from tar gets pruned → build catastrophe (false damage) |
| `modifiable_test_patterns` | tests agents may legally rewrite | missed → agent's legal test edit is discarded at capture |
| `prune_keep_list` | named fixture warts (exact paths, no globs) | missed → package-level collateral (navidrome `mock_library_service.go`: ~20 P2P unfairly lost) |
| `prune_extensions` | optional language narrowing | normally omit — the multi-language default plus the non-code-extension guard is the intended universal behavior |
| `residue_prune` | explicit enablement switch | absent or `false` → additive overlay (pruning disabled); set `true` only after the acceptance gate passes |

## Authoring procedure (per new range; all static, zero agent cost)

1. **Identify layout from the build system**, not from folder names: Cargo
   workspace members, `go.mod` + package dirs, Maven `<modules>`, `pyproject`/
   `setup.cfg` packages, `tsconfig`/`package.json` workspaces. `repo_src_dirs`
   = the union of source roots the agent is expected to own. Root build files
   are captured automatically (`ROOT_BUILD_FILES`) — do not list them.
2. **Enumerate test conventions for the ecosystem AND verify against the
   actual tree** (`git ls-tree -r`): `**/*_test.go` + `testdata/`; `tests/`,
   `benches/`, in-src `*_test.rs` files (list them explicitly — nushell has
   five); `**/src/test/**` + `*Test.java` + test/demo modules; `**/test_*.py`
   + `conftest.py` + `tests/` dirs; `*.test.*`/`*.spec.*` + `__mocks__` +
   `__snapshots__`. Convention lists are a starting point; the tree is the
   authority.
3. **Detect generated files by content, not by name**: grep the END tree for
   `Code generated by`, `DO NOT EDIT`, `@generated`, protoc/wire/bindgen
   headers; record patterns, not file lists.
4. **Inventory non-code assets inside src dirs** (`.sql`, `.pcss`, SPI
   declarations under `META-INF/`, `.mustache`, `.zip`, completions/man
   pages): they are protected by the code-extension whitelist automatically —
   the audit is to confirm none of them is actually agent-authoritative code
   in disguise (a `.sh` the agent must edit, say). If one is, that is a
   src_dirs/extension decision to record, not a keep-list entry.
5. **Recompute the keep-list** with the spec §3 discriminator, per milestone
   END tree: referenced ONLY by test files ∧ not mentioned in the SRS ∧
   fixture nature (a `testing.T` param / mock|fake|stub|fixture naming /
   `tests.*` embeds / "for testing" comments). Exact paths only. Known
   pitfalls: (a) run every GT-added-file scan with `--no-renames` — a
   trivially moved file is hidden by rename detection (go-zero
   `dbtest/sql.go` lesson); (b) **same-name milestone tags differ across a
   range's milestone images** (2026-07-11 nushell finding: two images'
   `core_development.1-end` trees differ by 5 files) — any offline analysis
   of a milestone's tree, and any V3 tar, must come from that milestone's
   OWN evaluation image, never a sibling's. Production evaluation is
   unaffected (it always runs inside the milestone's own image).
6. **Write the fields into the range `metadata.json`** in the data repo, one
   commit per range, citing this skill. Leave `residue_prune` absent or `false`
   while authoring; set it to `true` only after recording a successful
   acceptance run.

## Acceptance gate (must pass before the range goes live)

1. **V3 GT-as-agent invariant**: build a "perfect agent tar" (git archive the
   milestone END tree, filter through `should_include_in_snapshot` with the
   new config), run the full evaluator with pruning on. Required:
   `pruned_files_count == 0` and resolved (a resolution failure must be
   attributable to a non-prune cause, e.g. a known pathology-① island).
2. **V3b stays armed**: any protected-class path entering a prune set aborts
   the eval (`safety-abort`) — if the gate trips during acceptance, the
   config is wrong; fix the config, never relax the assertion.
3. **V4 A/B spot-check** where prior trial tars exist: evaluate ON vs OFF,
   attribute every per-test diff to "agent deletion no longer resurrected"
   (expected) — anything unattributable is a stop-and-investigate.
4. **Capture-side sidecar quiet**: on a pilot run, `uncommitted_*` /
   `committed_outside_*` counts should be zero or explainable; a systematic
   `committed_outside` signal means `repo_src_dirs` is too narrow.

Worked reference: `EvoClaw-log/residue_prune_verify/run4_universal/` (the
2026-07-11 universalization run: 5 ranges, V3 all-cover + V4 spot-check,
driver `run4.sh`, GT-tar builder pattern in the run's tooling).

## Output

A data-repo PR per range: metadata fields + a short authoring record (what
the build system says, which conventions were verified, keep-list
discriminator hits with evidence paths) + acceptance artifacts (V3 result,
V4 attribution table). Config goes to the data repo; anything that smells
like a new *mechanism* (a predicate the code layer should own) goes to an
issue on this repo instead.
